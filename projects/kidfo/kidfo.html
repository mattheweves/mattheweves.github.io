KIDFO

Store your child's basic information in the app. Name, Birthday, Eat Routine, Sleep Routine, No-Nos, and more.

SITTERS

Invite trusted parties to the app, and once they sign up and accept they'll be connected to your family and able to access vital information while under their care. No more worrying about writing a list out last minute. The information is stored in the app and always available to your care team.

KIDFO API


My first crack at an API from the ground up.
  • Validation errors can occur when creating or updating entities.
    • If so, it will produce the HTTP status code of 422 or unprocessable_entity.
  • Deletion requests will produce a status code of 204 or no_content when elements are successfully removed.
  • Resources not found will result in the HTTP status code of 404 or not_found. 

GET /kids
Description: this produces an array containing all of the kids the user has in a JSON format.

GET /kids/:id
Description: this retrieves a specific kid in a JSON format.

GET /sitters
Description: this produces an array containing all of the kids the user has in a JSON format.

GET /families
Description: this produces an array containing all of the kids the user has in a JSON format.

GET /family/:id
Description: this produces an array containing all of the kids the user has in a JSON format.

GET /user/:id
Description: this produces an array containing all of the kids the user has in a JSON format.

POST /kids
Description: create a new kid in the application.

The create action expects the following field to be present in the POST data of the request.

      {
        kid: {
          first name (middle name optional): “Bryar Louise”
        }
      }

PATCH /kids/:id
Description: update an existing kid in the database.

The update action expects the following field to be present in the form data of the request.

      {
        kid: {
          first name (middle name optional): “Bryar Louise”,
          all optional fields:

        }
      }

DELETE /kids/:id
Description: deletes the kid from the application.

POST /kids/:id/favorites
Description: add a favorite to the kid. This expects the following fields to be present in the POST data of the request:

      {
        favorite: {
          name: “Soccer”,
          category: “Activity”
        }
      }

DELETE /favorites/:id
Description: this removes the favorite with the provided id from the application.

Project Challenges:


JSON Token Authentication

It was important to break this into baby steps. You have to understand each step of the process, or it is VERY easy to get lost and difficult to troubleshoot. I used the Simple Token Authentication as it couples with Devise, a widely used authentication gem for Rails.

The gem installation is straightforward, but getting the flow right and token storage design on the React side dialed in takes a bit of planning, but very exciting once you get it working! 

const userAuth = () => {
              const session = {
                email: localStorage.getItem('email'),
                token: localStorage.getItem('token')
              };
              return { 'headers': {
                    'X-User-Email': session.email,
                    'X-User-Token': session.token
                    }
                  };
            };

export default userAuth;

Carrierwave Images / Amazon S3

This is a piece of cake if you are doing image uploading in a full Rails App. But when using Rails as the backend and trying to integrate with a different front-end framework, you need to pass that image from (in this case) your React Front End to the Rails API. How do you store an image? Well in short you have to: provide a form for image upload, convert it to a gigantic string (base64), send that to your API, have your API convert it back to an image, and finally upload that to your storage provider (mine was Amazon S3).


Rails Relationships

This is truly the perfect project for learning about parent and child relationships. Why? Because we are literally storing parent and child relationships. Here's how I broke it down. First was grasping the "has_many through" rails relationships, in the case of Kidfo it was:


belongs_to :family
has_many :kids, through: :family
Additionally, the idea of "sitters" whose class is User and role based permissions and functionality added complexity that took some good time and testing. 

#Sitters
has_many :familysitters, foreign_key: :sitter_id
has_many :families, through: :familysitters, source: :family


The rubydocs Physician/Appointment/Patient metaphor was very helpful in building this out.
